# Compile-time SQL assembly — macro pipeline for static query pieces

Addresses [#24](https://github.com/ragb/skunk-sharp/issues/24) (umbrella),
[#14](https://github.com/ragb/skunk-sharp/issues/14) (concrete implementation),
[#15](https://github.com/ragb/skunk-sharp/issues/15) (prerequisite hygiene).

## Context

Every skunk-sharp builder today compiles to a `skunk.AppliedFragment` by running a
`|+|` chain over many tiny `TypedExpr.render` calls. `AppliedFragment`'s SQL string
and argument list are assembled at runtime, even when every structural piece is a
constant known at compile time — column names (singleton strings), operator symbols
(`>=`, `=`, `LIKE`), `ORDER BY` direction (`ASC`/`DESC`), `LIMIT`/`OFFSET` integer
literals, projection lists, INSERT/UPDATE/DELETE skeletons. Only the user-supplied
parameter values (`18` in `u.age >= 18`) are genuinely runtime.

The user's constraint: **stable-shape queries must produce a compile-time-assembled
SQL string and bind only their parameters at runtime, but the DSL's user-facing
shape must stay the same**. Branchy runtime composition (different WHERE depending
on a feature flag, dynamic filter list from an HTTP request) must keep working —
that's the one genuine reason to compose at runtime.

skunk already provides the primitive we need: `skunk.sql"…"` is a Scala 3 quoted
macro that produces `Fragment[A]` with a compile-time-baked `parts` list
([skunk's StringContextOps.scala:162](file:///Users/ragb/src/skunk/modules/core/shared/src/main/scala-3/syntax/StringContextOps.scala#L162)).
So the job is: **change skunk-sharp's DSL methods so that each one emits a call to
skunk's `sql""` at the call site**, instead of constructing a runtime `TypedExpr`
tree that `.render`s piece-by-piece.

## Strategy: lift SQL strings to compile time; keep `AppliedFragment` as the glue

One-line principle: **every leaf builder call becomes an `inline` method that
delegates to a quoted macro which emits a `sql"…"` literal**. The builder still
carries an `AppliedFragment` internally; what changes is that each piece that
feeds the `|+|` chain is now a constant-SQL `Fragment[A].apply(a)` rather than a
runtime walk over a `TypedExpr` tree.

**Composition is still runtime.** skunk's `sql""` macro embeds inner fragments
as `Emb($f.parts)` — the `.parts` access is a runtime read, so `sql"$outer
$inner"` does not fold `inner`'s parts into `outer` at compile time
([StringContextOps.scala:99-108](file:///Users/ragb/src/skunk/modules/core/shared/src/main/scala-3/syntax/StringContextOps.scala#L99)).
The saving we get is per-piece: each `sql""` call bakes its own parts list into
a compile-time constant. A builder chain with five static pieces becomes five
interned arrays plus a runtime list-of-lists concat — vs today's ~30 runtime
`|+|` of tiny fragments. Order-of-magnitude fewer allocations, not zero.

**Full compile-time collapse (single interned string for the whole query)** is
a *separate harder problem*: it needs one macro owning the entire builder chain
so it can concatenate parts at expansion time. That's Phase 5 / future work —
not the POC. The POC targets "compile-time-baked pieces, runtime-concatenated".

Key property: **user code does not change**. `users.select.where(u => u.age >= 18)
.orderBy(_.created_at.desc).limit(10).compile.run(session)` reads the same;
`.where`, `.orderBy`, `.limit` become `inline` + macro-backed.

## Dynamism escape hatches

Three tiers, in order of preference:

1. **Static lambda** — `users.select.where(u => u.age >= 18)` — macro path.
2. **Branching over macro-emitted pieces** — `val q1 = if cond then q.where(…) else q`.
   Each branch's macro emits a pre-baked `AppliedFragment`; `if`/`match` picks one
   at runtime. Works today through the existing builder state and needs no extra
   API — just the macro path returns the same builder type regardless.
3. **Fully-runtime fragment** — `users.select.whereRaw(af: AppliedFragment)`. New
   explicit overload for the case where even the SQL structure is built at runtime
   (user-supplied filter list, ad-hoc extensions). Falls back to today's path.

Literal vs parameter: default is parameter (matches skunk's `sql""`). Opt in to
inlined literal with `lit(v)` — already exists as `TypedExpr.lit`; its semantics
become "emit the value as a SQL literal in the baked part" rather than "bind as
parameter".

## Phased rollout

### Phase 0 — Inlining audit (#15)

Prerequisite hygiene. No macro work yet.

- Run a pass over all `inline def` / `transparent inline def` in [modules/core/src/main/scala/skunk/sharp/](modules/core/src/main/scala/skunk/sharp/) and confirm each is actually inlined at call sites (check generated bytecode for a representative client).
- Find match-type stalls — the CLAUDE.md note about `AliasesOf` / `EffectiveViewsOf` is the canonical example.
- Make sure `TypedColumn`'s singleton name is accessible at compile time via a `transparent inline def columnName: String & Singleton`.
- Make `ColumnsView`'s structural access macro-pattern-friendly — the tree of `u.age` must be a shape the macro can reliably match on.

Deliverables: small PRs, each with before/after notes. No user-visible change.

### Phase 1 — POC: SELECT + single-predicate WHERE (#14 scope)

Smallest viable slice that proves the approach.

**Files to add / modify:**

- `modules/core/src/main/scala/skunk/sharp/internal/SqlMacros.scala` (new)
  - `whereMacro[Cols](f: Expr[ColumnsView[Cols] => Where]): Expr[AppliedFragment]`
  - Pattern-matches supported leaf shapes: `col === v`, `col !== v`, `col < v`, `<=`, `>`, `>=`, `like`, `in(seq)`, `isNull`, `isNotNull`.
  - For each matched shape: extract column singleton name at compile time, emit a `sql"\"$colName\" >= $encoder"`-style tree, return the applied fragment.
  - Unsupported shape → emit a `report.error` with a useful message **plus** fall back to the runtime `TypedExpr` path for the unmatched branch.
- [modules/core/src/main/scala/skunk/sharp/dsl/Select.scala](modules/core/src/main/scala/skunk/sharp/dsl/Select.scala)
  - `.where(f: ColumnsView[Cols] => Where)` becomes `inline def where(inline f: …): SelectBuilder[…]` → calls `${ SqlMacros.whereMacro('f) }`.
  - Add `.whereRaw(af: AppliedFragment)` overload for Tier-3 dynamism.
- [modules/core/src/main/scala/skunk/sharp/where/Ops.scala](modules/core/src/main/scala/skunk/sharp/where/Ops.scala)
  - Operator extension methods stay as runtime builders (they're the fallback path); the macro inspects the tree before those run.

**Verification (per #24 acid test):**

- `modules/core/src/test/scala/skunk/sharp/compiletime/WhereMacroSpec.scala` — compile-time assertion that `users.select.where(u => u.id === someUuid).compile.af.fragment.parts` is a single-element list with a constant string. Use `scala.compiletime.summonFrom` + `constValue` on a type-level SQL reflection, or simpler: inspect the quoted tree at macro expansion and emit the constant into a `val` we can query.
- Negative counterpart: the dynamic branch `if cond then q.where(…) else q` must produce a `parts` list whose contents depend on runtime, confirmed by bench harness (allocation per call > 0).
- Full existing suite passes unchanged.
- JMH micro-benchmark comparing allocations per `.compile` call (expect 1–2 allocations, down from 15–30).

### Phase 2 — Extend SELECT primitives

Once Phase 1 works:

- Projection list (`.select(u => (u.a, u.b))`) — macro emits the comma-joined column identifiers as a constant.
- `.orderBy(u => u.a.desc.nullsLast)` — direction, nulls clause are compile-time.
- `.limit(n)` / `.offset(n)` — macro captures the runtime int as a bound parameter OR inlines the literal when `n` is a `inline val`.
- `DISTINCT`, row-locking clauses (constants).

### Phase 3 — INSERT

- Column list, `VALUES` placeholders are compile-time from the named tuple's labels.
- Batch INSERT via `Reducible` stays runtime — the pre-baked single-row fragment is repeated.
- `RETURNING` clause: constant column list.
- `ON CONFLICT (…) DO UPDATE` — constraint column list and SET clause are compile-time; `EXCLUDED.col` refs are compile-time strings.

### Phase 4 — UPDATE, DELETE

- UPDATE `SET` clause: column names constant, RHS expressions macro-processed.
- `.patch(p: Product)` — named tuple labels are compile-time.
- DELETE: mostly `WHERE`, shared with Phase 1 infrastructure.

### Phase 5 — JOINs, subqueries, CTEs, window functions

- Multi-source builders: alias handling, `ON` predicates — all compile-time.
- Subqueries: `.asExpr`, `.in`, `Pg.exists` — macro recursion into the inner builder.
- CTE `WITH …` — structural.

## API changes (user-visible)

- `users.select.where(u => …)` — unchanged. Now inline + macro-backed.
- `users.select.whereRaw(af: AppliedFragment)` — **new**. Explicit dynamic escape
  hatch. `whereRaw` name signals that the user owns parameter binding.
- `lit(v)` semantics refined: "inline as SQL literal in the baked string" (was:
  "bind as parameter"). Default for `u.age >= 18` remains parameter binding.
- Everything else unchanged.

## Verification

1. **All existing unit + integration tests pass with no source changes** — the DSL surface is the same.
2. **Compile-time acid test** (issue #24 requirement):
   - `WhereMacroSpec` asserts that for a fully-static query, the produced `Fragment.parts` is `List(Left(sqlConst) :: Right(encoder) :: Left(sqlConst) :: Nil)` with the string literals verifiable via `constValue`.
   - Negative twin: dynamic query has its parts computed at runtime (verified by benchmarking allocation > 0 per call).
3. **JMH micro-benchmarks** under `modules/core/src/test/scala/…/bench/` (or a new `bench` sub-project):
   - Baseline: current `.compile.af` for representative query (SELECT + WHERE + ORDER + LIMIT).
   - Target: macro path — expect allocations per call to drop by an order of magnitude.
4. **`sbt compile` regression** on the `example` module — target < 30% (per #24).
5. **Bytecode inspection**: `javap` on a compiled client class should show the SQL string as a constant pool entry. Spot-check one file in Phase 1; automate in Phase 2+ if worth it.

## Risks

- **Compile-time cost blow-up** from heavy inlines / macros. Mitigation: one method at a time, measure after each.
- **Error-message regression** — macro errors are notoriously bad. Mitigation: every macro has an "unsupported pattern" path with a human-written `report.error`; unit tests that deliberately trigger those paths and assert on the message.
- **Code duplication** between macro path and runtime fallback. Accepted during rollout; consolidate once the macro covers everything. Keep the runtime `TypedExpr` tree around as the fallback — it's still needed for `.whereRaw` and for unsupported shapes.
- **Macro debuggability** — symptoms like "why didn't this inline" are annoying. Mitigation: a dev flag (`-Dskunk-sharp.debug-macros=true`) that makes macros log their expansion. Not shipped, just for development.

## Known design debt — consolidation required

PR #73 ships with **parallel `Typed*` classes** (`TypedSelectBuilder`,
`TypedDeleteReady`, `TypedUpdateReady`, `TypedInsertCommand`) alongside the
existing untyped builders. This is **not** the intended final design.

The correct design is a single builder per verb with `Args` as a type parameter:
`SelectBuilder[Ss, Args]`, `InsertCommand[Cols, Args]`, `UpdateReady[Cols, Args]`,
`DeleteReady[Cols, Args]`. Every method threads `Args`; operations that don't
capture runtime values leave it unchanged. Dynamism lives at an explicit
`.whereRaw(af: AppliedFragment)` escape hatch that widens `Args` to a wildcard.

Why it isn't in this PR: the consolidation ripples through ~30 files (every
consumer of `SelectBuilder[Ss]`: `Cte.scala`, `Join.scala`, `Compiled.scala`'s
`AsSubquery`, `SetOpQuery`, tests, integration suites) and needs each signature
threaded with `Args` or widened to `[Ss, ?]`. Attempted on a sub-branch this
session; reverted because the WIP state wasn't landing in the session budget.

Consolidation checklist (concrete next session):
- `SelectBuilder[Ss, Args]` — thread Args through `.where`, `.having`,
  `.orderBy`, `.limit`, `.offset`, `.distinctRows`, `.distinctOn`, all `*Join`
  methods, locking. Change storage of `whereOpt` / `havingOpt` to typed
  `Fragment[?] + args` pair.
- `.whereRaw(af)` / `.havingRaw(af)` escape hatches returning
  `SelectBuilder[Ss, ?]`.
- Operators (`===`, `>=`, …) always return `TypedWhere[Stripped[T]]`; the
  non-`TypedColumn` LHS branch uses `Encoder.contramap` to bake the LHS's
  pre-bound args and produce a `TypedWhere[Stripped[T]]` whose fragment handles
  both positions.
- `Insert`/`Update`/`Delete` builders same treatment — Args from captured
  values, `.where` overload threads, `.returning` variants parameterise output.
- Delete `TypedSelectBuilder`, `TypedDeleteReady`, `TypedUpdateReady`,
  `TypedInsertCommand`, `TypedSetAssignment`, `TypedReturning`.
- Mass-update `CompiledQuery[?, R]` / `SelectBuilder[Ss, ?]` in tests and
  subquery typeclass instances.
- `MutationReturning[R]` becomes `MutationReturning[Args, R]`.

Estimated effort: one focused day. Not this session.

## Status (as of April 24, 2026)

PR #73 now carries a **functioning typed pipeline** for SELECT + DELETE with
`Args` threaded at the type level. Explicit typed macros (`SqlMacros.infixTyped`)
produce `TypedWhere[A]`; `SelectBuilder.where` / `DeleteBuilder.where`'s typed
overloads pick it up and return typed builders.

**Working typed end-to-end:**

```scala
val pred = SqlMacros.infixTyped[Int, Int](">=", u.age, 18)
val q    = users.select
  .where(_ => pred)                     // → TypedSelectBuilder[Ss, Int]
  .groupBy(u => u.email)                // Args passthrough
  .having(_ => otherPred)                // Args = (Int, OtherA)
  .orderBy(u => u.email.asc)             // Args passthrough
  .limit(10)
  .compile                              // → CompiledQuery[(Int, OtherA), NamedRow]

users.delete.where(_ => pred).compile   // → CompiledCommand[Int]
```

**`TypedSelectBuilder[Ss, Args]` — full typed SELECT chain:**
`.where` (AND-combined), `.groupBy`, `.having` (combined-encoder product),
`.orderBy`, `.limit`, `.offset`, `.distinctRows`, row-level locking. Each
method knows whether it captures runtime params; `Args` is threaded
accordingly.

**`TypedDeleteReady[Cols, Name, Args]` — typed DELETE:**
`.where` (repeatable, AND-combined). `.compile` → `CompiledCommand[Args]`.

**Pending for a fully all-or-nothing typed experience:**
- Typed UPDATE — `.set(u => u.col := value)` captures values; each assignment
  extends Args. Staged builder pattern needs typed equivalents for
  `UpdateBuilder` → `UpdateWithSet` → `UpdateReady`.
- Typed INSERT — row's runtime values are the Args tuple; batch insert with
  `Reducible` stays on typed path via encoder replication.
- Typed RETURNING on INSERT/UPDATE/DELETE — `.returning` / `.returningAll`
  taking the typed builder through to `CompiledQuery[Args, R]`.
- Multi-source forms: `DELETE ... USING`, `UPDATE ... FROM`, JOIN-sourced
  SELECT typed path.
- `.select(projection)` on `TypedSelectBuilder` — projections that introduce
  params (`Pg.greatest(col, lit(100))`) would need Args from their expressions;
  most projections are pure column refs with no capture.

## Path to the ergonomic typed end — exploration notes

PR #73 landed the foundations:
  • `CompiledQuery[Args, R]` / `CompiledCommand[Args]` as real type parameters.
  • `TypedWhere[Args] <: Where` with `.&&` combinator.
  • `SqlMacros.infixTyped` / `infix2Typed` / `postfixTyped` — explicit typed macros.
  • `SelectBuilder.where[A](f: SelectView => TypedWhere[A]): TypedSelectEnd[Ss, A]`
    overload + `TypedSelectEnd.compile: CompiledQuery[Args, R]`.

The ergonomic path — `users.select.where(u => u.age === 18).compile` yielding
`CompiledQuery[Int, Row]` without `SqlMacros.infixTyped` gymnastics — ran into
Scala 3 type-system limits that I explored and aborted:

1. **Two extensions, one name, different receivers (TypedExpr vs TypedColumn).**
   Scala 3's extension specificity does NOT prefer the more-specific-receiver
   extension when the names overlap. The TypedExpr `===` wins; the TypedColumn
   one is ignored. Unique names (e.g. `eqT`) DO work but are ugly.
2. **`transparent inline` cascade through `valOp` / `SqlMacros.infix`.**
   The macro emits `TypedWhere[S]` for the TypedColumn branch but `Where` for
   the fallback. The declared return type at every level stays `Where`. Scala
   does not re-narrow via transparent inline when the macro has branching
   return shapes.
3. **Match-type dispatch via `OpResult[L, S]` on the extension receiver type.**
   Works for a single operator in isolation (`col === 42` → `TypedWhere[Int]`)
   but cascades broke downstream code: `.where` now resolves to the typed
   overload for ALL existing lambda bodies, returning `TypedSelectEnd[...]`
   instead of `SelectBuilder[...]`. `CteSuite`'s `cte("x", posts.select.where(…))`
   fails because `cte(…)` has no overload for `TypedSelectEnd`. Fixing the
   cascade means `TypedSelectEnd` must expose every method on `SelectBuilder`
   (or `SelectBuilder` must thread Args through its whole surface).

The realistic path to the full ergonomic typed end:

- Add an `Args` type parameter to `SelectBuilder[Ss, Args]` (and all other
  builders) with `Void` as the "no args captured yet" value. Every method
  threads Args on.
- `.where` untyped overload: `SelectBuilder[Ss, Args]` → `SelectBuilder[Ss, Args]`
  (unchanged args, since the untyped lambda erases to Where).
- `.where` typed overload: `SelectBuilder[Ss, Args]` → `SelectBuilder[Ss, ConcatArgs[Args, A]]`
  — when the lambda returns `TypedWhere[A]`.
- `&&` / `||` on `TypedWhere` use match-type dispatch: two typed operands →
  typed result; any existential operand → existential result.
- Every DSL verb that captures runtime values (`.having`, `.limit`, `.offset`,
  `.set` on UPDATE, etc.) threads Args through the same way.
- `Insert` / `Update` / `Delete` / `ProjectedSelect` get the same type-parameter
  surgery.
- JOINs' multi-source builders also thread Args.

Estimated effort: 2–4 days of focused work across every builder / operator file.
Not one-more-hour.

## Non-goals (explicitly)

- Rewriting the DSL's user-facing shape.
- Deprecating `AppliedFragment` as the internal representation in v1. That's a possible future refactor (tracked implicitly in #24's "Type-level query skeletons" point), not part of this work.
- Hand-writing SQL. Users still see the typed builder.
- Cross-backend support.
